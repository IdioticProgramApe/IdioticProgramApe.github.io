## Rust vs. C++ vs. Python

### Typing

|                    | Statically Typed             | Dynamically Typed        |
| ------------------ | ---------------------------- | ------------------------ |
| **Strongly Typed** | C/C++, C#, Rust, Java, Swift | Python, Ruby, TypeScript |
| **Weakly Typed**   | *C/C++*                      | JavaScript, PHP, Perl    |

- **Statically** Typed Languages: Languages where variable types are checked and enforced at compile time (before the program runs).
- **Dynamically** Typed Languages: Languages where variable types are checked at runtime (while the program is running).
- **Strongly** Typed Languages: Languages that enforce strict adherence to type rules, disallowing implicit or automatic type conversions between incompatible types.
- **Weakly** Typed Languages: Languages that allow more flexible type conversions, often performing implicit coercion between different types during operations.

### Object-Oriented Programming

The following table lists the composites of an entity in Rust and its equivalents in both C++ and Python.

| Rust[^1]                 | C++ Equivalent                                 | Python[^2] Equivalent            |
| ------------------------ | ---------------------------------------------- | -------------------------------- |
| `struct`                 | `struct`/`class` with only data                | `class` with only data           |
| inherent implementations | non virtual function definitions               | function definitions (up to dev) |
| `trait`                  | interface w/o default function implementations | abstract class / protocol        |
| trait implementations    | virtual function overrides                     | function overrides (up to dev)   |

[^1]: Rust separates data and methods into `struct` and `impl` blocks respectively, `trait` only contains functions (declaration w/o default implementation).

[^2]: Python is a dynamic, strong typed language, and follows duck typing standards.

#### Derive Macro

Derive macro is used similarly as a decorator in python, which accepts traits as input params, Derive will pick up the traits and auto-generate the implementation for the decorated struct, i,.e.:

```rust
#[derive(Debug, PartialEq)]
struct Ticket {
    title: String,
    description: String,
    status: String,
}
```

here the derived `PartialEq` is equivalent to (using [`cargo expand`](https://crates.io/crates/cargo-expand) to expand the generated code):

```rust
#[automatically_derived]
impl ::core::cmp::PartialEq for Ticket {
    #[inline]
    fn eq(&self, other: &Ticket) -> bool {
        self.title == other.title && self.description == other.description
            && self.status == other.status
    }
}
```

### Generic Programming

a.k.a, the templates in C++, however, in rust, this is done through trait as well, we can explicitly state the trait bounds or inline them if they are simple enough, consider this trait:

```rust
trait IsEven { fn is_even(&self) -> bool; }
```

we have the generic function `print_if_even`:

```rust
// explicit trait bound
fn print_if_even<T>(n: T)
where
	T: IsEven + Debug
{
    if n.is_even() { println!("{n:?} is even"); }
}

// inline trait bound
fn print_if_even<T: IsEven + Debug>(n: T) { ... }
```

#### Deref Coercion[^3]

Given 2 Types `T` and `U`, we can tell the compiler `&T` and `&U` are interchangeable somewhat by implementing `Deref<Target=U> for T`:

- `&T` can be converted into `&U` implicitly
- call all the method `T::func(&self)` on `U`

Example: `&String` -> `&str`

[^3]: don't abuse deref coercion, a common and safe use case of this feature is the smart pointers.

#### From & Into

```rust
pub trait From<T>: Sized { fn from(value: T) -> Self; }
pub trait Into<T>: Sized { fn into(self) -> T; }
```

- supertrait / subtrait: `From` and `Into` are subtraits of `Sized`, `Sized` is the supertrait of `From` and `Into`.

- implicit trait bounds: `<T>` is equivalent to `<T: Sized>`, however if we want the generic type to accept DST (Dynamic Sized Type) as well, can use `<T: ?Sized>`:

  - it means, the type `T` may or may not be `Sized`, and the syntax only works on `Sized` trait

- dual traits: they are tied with a [**blanket implementation**](https://stackoverflow.com/questions/73861891/what-are-blanket-implementations-in-rust):

  ```rust
  impl<T, U> Into<U> for T
  where
  	U: From<T>,
  {
      fn into(self) -> U { U::from(self) }
  }
  ```
