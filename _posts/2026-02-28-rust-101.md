## Typing

|                    | Statically Typed             | Dynamically Typed        |
| ------------------ | ---------------------------- | ------------------------ |
| **Strongly Typed** | C/C++, C#, Rust, Java, Swift | Python, Ruby, TypeScript |
| **Weakly Typed**   | *C/C++*                      | JavaScript, PHP, Perl    |

- **Statically** Typed Languages: Languages where variable types are checked and enforced at compile time (before the program runs).
- **Dynamically** Typed Languages: Languages where variable types are checked at runtime (while the program is running).
- **Strongly** Typed Languages: Languages that enforce strict adherence to type rules, disallowing implicit or automatic type conversions between incompatible types.
- **Weakly** Typed Languages: Languages that allow more flexible type conversions, often performing implicit coercion between different types during operations.

### Derive Macro

Derive macro is used similarly as a decorator in python, which accepts traits as input params, Derive will pick up the traits and auto-generate the implementation for the decorated struct, i,.e.:

```rust
#[derive(Debug, PartialEq)]
struct Ticket {
    title: String,
    description: String,
    status: String,
}
```

here the derived `PartialEq` is equivalent to (using [`cargo expand`](https://crates.io/crates/cargo-expand) to expand the generated code):

```rust
#[automatically_derived]
impl ::core::cmp::PartialEq for Ticket {
    #[inline]
    fn eq(&self, other: &Ticket) -> bool {
        self.title == other.title && self.description == other.description
            && self.status == other.status
    }
}
```

## Generic Programming

a.k.a, the templates in C++, however, in rust, this is done through trait as well, we can explicitly state the trait bounds or inline them if they are simple enough, consider this trait:

```rust
trait IsEven { fn is_even(&self) -> bool; }
```

we have the generic function `print_if_even`:

```rust
// explicit trait bound
fn print_if_even<T>(n: T)
where
	T: IsEven + Debug
{
    if n.is_even() { println!("{n:?} is even"); }
}

// inline trait bound
fn print_if_even<T: IsEven + Debug>(n: T) { ... }
```

### Deref Coercion (Associated Type)[^3]

Given 2 Types `T` and `U`, we can tell the compiler `&T` and `&U` are interchangeable somewhat by implementing `Deref<Target=U> for T`:

- `&T` can be converted into `&U` implicitly
- call all the method `T::func(&self)` on `U`

Example: `&String` -> `&str`

[^3]: don't abuse deref coercion, a common and safe use case of this feature is the smart pointers.

### From & Into (Generic Type)

```rust
pub trait From<T>: Sized { fn from(value: T) -> Self; }
pub trait Into<T>: Sized { fn into(self) -> T; }
```

- supertrait / subtrait: `From` and `Into` are sub-traits of `Sized`, `Sized` is the super-trait of `From` and `Into`.

- implicit trait bounds: `<T>` is equivalent to `<T: Sized>`, however if we want the generic type to accept DST (Dynamic Sized Type) as well, can use `<T: ?Sized>`:

  - it means, the type `T` may or may not be `Sized`, and the syntax only works on `Sized` trait

- dual traits: they are tied with a [**blanket implementation**](https://stackoverflow.com/questions/73861891/what-are-blanket-implementations-in-rust):

  ```rust
  impl<T, U> Into<U> for T
  where
  	U: From<T>,
  {
      fn into(self) -> U { U::from(self) }
  }
  ```


### Generics & Associated Types

- An associated type is uniquely determined by the trait implementation, which mean it cannot be implemented more than once (as the `Target` type in trait `Deref`)
- Generics allow multiple trait implementations for a type, as long as the the input type is different

```rust
pub trait Add<RHS = Self> {
    type Output;
    fn add(self, rhs: RHS) -> Self::Output;
}

impl Add<u32> for u32 {
    type Output = u32;
    fn add(self, rhs: u32) -> u32 { ... }
}

impl Add<&u32> for u32 {
    type Output = u32;
    fn add(self, rhs: &u32) -> u32 { ... }
}
```

The trait `Add` above mixed both generics and associated types, it allows 2 implementations for u32 with different generic input type `u32` and `&u32`, for each implementation, they have their own unique associated type `u32`, the impls above allow the following code to compile:

```rust
let x = 5u32 + &5u32 + 5u32;
```

Associated type is a new feature in Rust, without having proper analogies in C++ or in python.

### Clone Trait

- applicable when we need to use the value after it’s consumed beforehand
- `Clone` is a std trait, method `clone`, takes a reference to `self` and returns a new **owned** instance of the same type. 

```rust
let mut s = String::from("Hello");
let t = s.clone();
```

- memory intepretation

```
               s                               t
      +-----+-----+----------+        +-----+-----+----------+
stack | ptr | len | capacity |        | ptr | len | capacity |
      |  |  |  5  |    5     |        |  |  |  5  |    5     |
      +--|--+-----+----------+        +--|--+-----+----------+
         |                               |                    
         v                               v                    
       +---+---+---+---+---+           +---+---+---+---+---+  
heap   | H | e | l | l | o |           | H | e | l | l | o |  
       +---+---+---+---+---+           +---+---+---+---+---+  
```

- `Clone` implementation: it clones each fields of the decorated type individually and constructs a new instance of same type using the cloned fields.

```rust
#[derive(Clone)]
struct ClonableStruct { fields... }
```

### Copy Trait

- a marker trait (same as `Sized`)
- `Copy` is a sub-trait of `Clone`, and if a type implements `Copy`, then rust create a new instance **implicitly** (by performing a bitwise copy)

```rust
let s: u32 = 5;
consumer(s); 	// fn consumer(s: u32) { ... }, no clone before hands
let t = s + 1;	// no error
```

- copy implies auto clone, but not equivalents to it, type requirements to implement `Copy`:
  - type must implement `Clone`, since `Copy` does create new instance implicitly
  - type doesn’t manage any *additional* resources (e.g. heap memory, file handles...)
  - type is **NOT** a mutable ref (`&mut T`)

- usually no need implement `Copy` manually:

```rust
#[derive(Copy, Clone)]
struct CopyableStruct { field: u32, }
```

### Drop Trait

- drop function reclaims the memory occupied by the type (`std::mem::size_of` bytes)
- drop function cleans up any additional resources that the value might be managing (e.g. the heap buffer of a `String` instance)

```rust
pub trait Drop { fn drop(&mut self); }
```

- allows to define ***additional*** cleanup logic for the type (on top of what compiler does automatically).
- is executed when the value goes out of scope.
- implies a type has additional resources (checked by `Copy` trait), even if no explicit implementation in the `drop` function.

### Pattern Comparison

|                 | Rust                    | C                   | C++                 | Python             |
| --------------- | ----------------------- | ------------------- | ------------------- | ------------------ |
| **Supported**   | ✓                       | ⍻ (integer only)    | ⍻ (integer only)    | ⍻ (Python 3.10+)   |
| **Exhaustive**  | ✓                       | ✗                   | ✗                   | ✗                  |
| **Syntax**      | `match enumvar { ... }` | `switch... case...` | `switch... case...` | `match... case...` |
| **Catch-All**   | `_ => ...`              | `default: ...`      | `default: ...`      | `case _: ...`      |
| **Fall Though** | ✗                       | ✓                   | ✓                   | ✗                  |

- exhaustiveness: all enum variants must be handle, otherwise will lead to a compile error
- catch-all: not a best practice
- C/C++ pattern matching falls through without `break`, make sure break after a case is properly handled

#### Variant Data

```rust
enum Status {
    ToDo,
    InProgress { assigned_to: String },
    Done,
}

fn print_status(status: &Status) -> () {
    match status {
        Status::InProgress { assigned_to: person } => {
            println!("Assigned to: {}", person);
        },
        Status::ToDo | Status::Done => {
            println!("Todo or done");
        }
    }
}
```

- only rust supports attaching data to variant naturally, use pattern matching to access the variant data
- `assigned_to: person`: is called data binding, without `: person`, we still can use `assigned_to`

#### if let / let else

``` rust
impl Ticket {
    fn assigned_to_v1(&self) -> &str {
        if let Status::InProgress { assigned_to } = &self.status {
            assigned_to
        } else {
            panic!( ... );
        }
    }
    
    fn assigned_to_v2(&self) -> &str {
        let Status::InProgress { assigned_to } = &self.status else {
            panic!( ... );
        };
        assigned_to
    }
}
```

- `if let` allows to match a single variant of an enum, `let else` achieves the same thing, but avoid a level of indentation
  - notice that the return type is `&str`which is coerced from `&String`, since pattern match is against `&Status`, this reference will be passed to all of its sub-elements.
- don’t overdo it, `match` will always a viable option

### Special Enums

#### Option: Nullable Values

```rust
enum Option<T> { Some(T), None, }
```

- `Some` is a tuple-like variant, which holds unnamed fields, useful to act like a wrapper for single field

#### Result: Error Handling

```rust
enum Result<T, E> { Ok(T), Err(E), }
```

- Generic type `T` represents the output of the successful operation

- `E` represents the error generated by the failed operation, which is regarded as recoverable errors unlike what `panic!` represents

- `Result` enum as return type underlines that the function can fail

- `?` operation: a shorthand for propagating errors, often used in the end of a function which return a `Result` object:

  ```rust
  use std::fs::File;
  let mut file = File::open("file.txt")?;
  // equivalent to
  let mut file = match File::open("file.txt") {
      Ok(file) => file,
      Err(e) => { return Err(e) }
  }
  ```

  - make sure the caller also returns a `Result` object can the Error type can be constructed from the callee’s error

#### Error Reporting: Error Trait

- `Error` trait is a sub-trait of `Debug` and `Display`
- `Debug` can be implemented through `#[derive(Debug)]`, as a low-level feature
- `Display` must be implemented manual by `impl std::fmt::Display for ...`
- once `Debug` and `Display` implemented for type `T`, `Error` can be achieved by `impl std::fmt::Error for T { ... }`
- `Error` trait has a method call `source` to access the error cause if any, return type `Option<&(dyn Error + 'static)>`
  - `dyn Error`: trait object, implies any type that implements the `Error` trait
  - `‘static`: lifetime specifier, implies reference is statically valid throughout the entire program execution


## Packaging System

- **package** is defined in `[package]` section in Cargo.toml as known as the manifest file for a rust package

- **crate** is inside a package, one package can hold multiple crates, also known as targets

  - **binary crate**: can be compiled into an executable file, underlined by *src/main.rs*
  - **library crate**: cannot be executed on its own often work as dependency for other binaries, underlined by *src/lib.rs*, only one is allowed in one package

- **dependencies** is defined in `[dependencies]` section in Cargo.toml, as a form of a list of name, version pairs

  - dependencies will be looked up from [crates.io](https://crates.io/), and downloaded during **cargo build**

  - manifest unchanged + *Cargo.lock* => skip dependency resolution, unless use **cargo update**, *Cargo.lock* should be version controlled

  - local dependencies are also allowed

    ```toml
    [dependencies]
    local-lib = { path = "../local-lib" }
    ```

- **dev dependencies** is defined in `[dev-dependencies]` section in Cargo.toml, which only needed for development when running **cargo test**

## Containers

### Arrays

|                 | Rust Array                                                   | C Array                                                  | C++ Array                                                | Python List / Array                    |
| --------------- | ------------------------------------------------------------ | -------------------------------------------------------- | -------------------------------------------------------- | -------------------------------------- |
| **Declaration** | `let arr: [i32; 3] = [1, 2, 3];`<br />`let arr: [i32; 3] = [1; 3];` | `int arr[3] = {1, 2, 3};`<br />`int arr[3] = {1, 1, 1};` | `int arr[3] = {1, 2, 3};`<br />`int arr[3] = {1, 1, 1};` | `arr = [1, 2, 3]`<br />`arr = [1] * 3` |
| **Fixed size**  | ✓                                                            | ✓                                                        | ✓                                                        | ✗                                      |
| **Homogeneity** | ✓                                                            | ✓                                                        | ✓                                                        | ⍻                                      |
| **Access**      | `arr[index]`<br />`arr.get(index)` avoid panicking           | `arr[index]`                                             | `arr[index]`                                             | `arr[index]`                           |
| **Length**      | `arr.len()`                                                  | `sizeof(arr)/sizeof(arr[0])`                             | `std::size(arr)`                                         | `len(arr)`                             |

- memory allocated on stack
- sized-fixed at compile time, unlike python list, which is dynamic sized
- all elements must have the same type
- 0 indexed, the same as c, c++ and python, index must be of type `usize`
- panic on out of bound access, use `get` method to avoid, instead of panicking, it returns an `None` object wrapped in an `Option<&T>`

### Vectors

|                       | Rust `Vec<T>`                                                | C++ (`std::vector<T>`)         | Python (list)                 |
| --------------------- | ------------------------------------------------------------ | ------------------------------ | ----------------------------- |
| **Syntax**            | `let mut v = Vec::new();`<br />`let mut v = Vec::with_capacity(3);`<br />`vec![1,2,3]` | `std::vector<int> vec{1,2,3};` | `vec = [1, 2, 3]`             |
| **Typed**             | ✓                                                            | ✓                              | ⍻ (dynamically typed)         |
| **Memory management** | Ownership                                                    | RAII                           | GC                            |
| **Exception**         | ✓ (panics on OOB)                                            | ✓ (throws on OOB)              | ✓ (raises on OOB)             |
| **Length**            | `.len()`                                                     | `.size()`                      | `len()`                       |
| **Capacity**          | `.capacity()`                                                | `.capacity()`                  | n/a                           |
| **Resize**            | `.resize()`                                                  | `.resize()`                    | n/a                           |
| **Index access**      | `vec[index]`                                                 | `vec[index]` or `.at(index)`   | `list[index]`                 |
| **Append**            | `vec.push(value)`                                            | `vec.push_back(value)`         | `list.append(value)`          |
| **Removal**           | `vec.pop()`, `.remove(index)`                                | `vec.pop_back()`, `.erase()`   | `list.pop()`, `list.remove()` |
| **Iteration**         | `for item in &vec { }`                                       | Range-for or iter-based loops  | `for item in list:`           |

- dynamic array is not naturally supported in C, need to carefully manage the memory to perform all ops
- `Vec<T>` keeps tracking the pointer pointing to the reserved heap memory, the length of the vector and the capacity
- it’s used to implement `String` object, therefore same memory layout (`Vec<u8>`)
- `Vec` resizing can be expensive, if it happens, memory allocator will find another larger heap region, copy all the old data over and deallocate the old memory block
  - pre-allocate memory to reduce resizing as in c++
